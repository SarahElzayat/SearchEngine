      <p><b>STL</b> is a <a href="/wiki/File_format" title="File format">file format</a> native to the <a href="/wiki/Stereolithography" title="Stereolithography">stereolithography</a> <a href="/wiki/Computer-aided_design" title="Computer-aided design">CAD</a> software created by <a href="/wiki/3D_Systems" title="3D Systems">3D Systems</a>.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> STL has several <a href="/wiki/Backronym" title="Backronym">backronyms</a> such as "<b>S</b>tandard <b>T</b>riangle <b>L</b>anguage" and "<b>S</b>tandard <a href="/wiki/Tessellation" title="Tessellation"><b>T</b>essellation</a> <b>L</b>anguage".<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> This file format is supported by many other software packages; it is widely used for <a href="/wiki/Rapid_prototyping" title="Rapid prototyping">rapid prototyping</a>, <a href="/wiki/3D_printing" title="3D printing">3D printing</a> and <a href="/wiki/Computer-aided_manufacturing" title="Computer-aided manufacturing">computer-aided manufacturing</a>.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> STL files describe only the surface geometry of a three-dimensional object without any representation of color, texture or other common CAD model attributes. The STL format specifies both <a href="/wiki/ASCII" title="ASCII">ASCII</a> and <a href="/wiki/Binary_file" title="Binary file">binary</a> representations. Binary files are more common, since they are more compact.<sup id="cite_ref-burns_7-0" class="reference"><a href="#cite_note-burns-7">[7]</a></sup> </p>
      <p>An STL file describes a raw, unstructured <a href="/wiki/Triangulation_(geometry)" title="Triangulation (geometry)">triangulated</a> surface by the <a href="/wiki/Unit_vector" title="Unit vector">unit</a> <a href="/wiki/Surface_normal" class="mw-redirect" title="Surface normal">normal</a> and vertices (ordered by the <a href="/wiki/Right-hand_rule" title="Right-hand rule">right-hand rule</a>) of the triangles using a three-dimensional <a href="/wiki/Cartesian_coordinate_system" title="Cartesian coordinate system">Cartesian coordinate system</a>. In the original specification, all STL coordinates were required to be positive numbers, but this restriction is no longer enforced and negative coordinates are commonly encountered in STL files today. STL files contain no scale information, and the units are arbitrary.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> </p>

      <h2><span class="mw-headline" id="ASCII_STL">ASCII STL</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=1" title="Edit section: ASCII STL">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>An ASCII STL file begins with the line </p>
      <pre>solid <i>name</i>
</pre>
      <p>where <i>name</i> is an optional string (though if <i>name</i> is omitted there must still be a space after solid). The file continues with any number of triangles, each represented as follows: </p>
      <pre>facet normal <i>n<sub>i</sub> n<sub>j</sub> n<sub>k</sub></i>
    outer loop
        vertex <i>v</i>1<sub><i>x</i></sub> <i>v</i>1<sub><i>y</i></sub> <i>v</i>1<sub><i>z</i></sub>
        vertex <i>v</i>2<sub><i>x</i></sub> <i>v</i>2<sub><i>y</i></sub> <i>v</i>2<sub><i>z</i></sub>
        vertex <i>v</i>3<sub><i>x</i></sub> <i>v</i>3<sub><i>y</i></sub> <i>v</i>3<sub><i>z</i></sub>
    endloop
endfacet
</pre>
      <p>where each <i>n</i> or <i>v</i> is a <a href="/wiki/Floating-point_number" class="mw-redirect" title="Floating-point number">floating-point number</a> in sign-<a href="/wiki/Significand" title="Significand">mantissa</a>-"e"-sign-<a href="/wiki/Exponent" class="mw-redirect" title="Exponent">exponent</a> format, e.g., "2.648000e-002". The file concludes with </p>
      <pre>endsolid <i>name</i>
</pre>
      <div class="thumb tright">
       <div class="thumbinner" style="width:222px;">
        <span class="mw-3d-wrapper" data-label="3D"><a href="/wiki/File:Sphericon.stl" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/b1/Sphericon.stl/220px-Sphericon.stl.png" decoding="async" width="220" height="165" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/b1/Sphericon.stl/330px-Sphericon.stl.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/b1/Sphericon.stl/440px-Sphericon.stl.png 2x" data-file-width="5120" data-file-height="2880"></a></span>
        <div class="thumbcaption">
         <div class="magnify">
          <a href="/wiki/File:Sphericon.stl" class="internal" title="Enlarge"></a>
         </div>An example <a class="external text" href="https://upload.wikimedia.org/wikipedia/commons/b/b1/Sphericon.stl">ASCII STL</a> of a <a href="/wiki/Sphericon" title="Sphericon">sphericon</a>
        </div>
       </div>
      </div>
      <p>The structure of the format suggests that other possibilities exist (e.g., facets with more than one "loop", or loops with more than three vertices). In practice, however, all facets are simple triangles. </p>
      <p>White space (spaces, tabs, newlines) may be used anywhere in the file except within numbers or words. The spaces between "facet" and "normal" and between "outer" and "loop" are required.<sup id="cite_ref-burns_7-1" class="reference"><a href="#cite_note-burns-7">[7]</a></sup> </p>
      <h2><span class="mw-headline" id="Binary_STL">Binary STL</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=2" title="Edit section: Binary STL">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>Because ASCII STL files can be very large, a binary version of STL exists. A binary STL file has an 80-character header (which is generally ignored, but should never begin with "solid" because that may lead some software to assume that this is an ASCII STL file). Following the header is a 4-byte <a href="/wiki/Little-endian" class="mw-redirect" title="Little-endian">little-endian</a> unsigned integer indicating the number of triangular facets in the file. Following that is data describing each triangle in turn. The file simply ends after the last triangle. </p>
      <p>Each triangle is described by twelve 32-bit floating-point numbers: three for the normal and then three for the X/Y/Z coordinate of each vertex – just as with the ASCII version of STL. After these follows a 2-byte ("short") unsigned integer that is the "attribute byte count" – in the standard format, this should be zero because most software does not understand anything else.<sup id="cite_ref-burns_7-2" class="reference"><a href="#cite_note-burns-7">[7]</a></sup> </p>
      <p>Floating-point numbers are represented as <a href="/wiki/IEEE_floating-point" class="mw-redirect" title="IEEE floating-point">IEEE floating-point</a> numbers and are assumed to be <a href="/wiki/Little-endian" class="mw-redirect" title="Little-endian">little-endian</a>, although this is not stated in documentation. </p>
      <pre>UINT8[80]    – Header                 -     80 bytes
UINT32       – Number of triangles    -      4 bytes
</pre>
      <pre>foreach triangle                      - 50 bytes:
    REAL32[3] – Normal vector             - 12 bytes
    REAL32[3] – Vertex 1                  - 12 bytes
    REAL32[3] – Vertex 2                  - 12 bytes
    REAL32[3] – Vertex 3                  - 12 bytes
    UINT16    – Attribute byte count      -  2 bytes
end
</pre>
      <h2><span class="mw-headline" id="Color_in_binary_STL">Color in binary STL</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=3" title="Edit section: Color in binary STL">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>There are at least two non-standard variations on the binary STL format for adding color information: </p>
      <ul>
       <li>The VisCAM and SolidView software packages use the two "attribute byte count" bytes at the end of every triangle to store a 15-bit <a href="/wiki/RGB" class="mw-redirect" title="RGB">RGB</a> color:
        <ul>
         <li>bits 0 to 4 are the intensity level for blue (0 to 31),</li>
         <li>bits 5 to 9 are the intensity level for green (0 to 31),</li>
         <li>bits 10 to 14 are the intensity level for red (0 to 31),</li>
         <li>bit 15 is 1 if the color is valid, or 0 if the color is not valid (as with normal STL files).</li>
        </ul></li>
       <li>The Materialise Magics software uses the 80-byte header at the top of the file to represent the overall color of the entire part. If color is used, then somewhere in the header should be the <a href="/wiki/ASCII" title="ASCII">ASCII</a> string "COLOR=" followed by four bytes representing red, green, blue and <a href="/wiki/Alpha_channel" class="mw-redirect" title="Alpha channel">alpha channel</a> (transparency) in the range 0–255. This is the color of the entire object, unless overridden at each facet. Magics also recognizes a material description; a more detailed surface characteristic. Just after "COLOR=RGBA" specification should be another ASCII string ",MATERIAL=" followed by three colors (3×4 bytes): first is a color of <a href="/wiki/Diffuse_reflection" title="Diffuse reflection">diffuse reflection</a>, second is a color of <a href="/wiki/Specular_highlight" title="Specular highlight">specular highlight</a>, and third is an <a href="/wiki/Shading#Ambient_lighting" title="Shading">ambient light</a>. Material settings are preferred over color. The per-facet color is represented in the two "attribute byte count" bytes as follows:
        <ul>
         <li>bits 0 to 4 are the intensity level for red (0 to 31),</li>
         <li>bits 5 to 9 are the intensity level for green (0 to 31),</li>
         <li>bits 10 to 14 are the intensity level for blue (0 to 31),</li>
         <li>bit 15 is 0 if this facet has its own unique color, or 1 if the per-object color is to be used.</li>
        </ul></li>
      </ul>
      <p>The red/green/blue ordering within those two bytes is reversed in these two approaches – so while these formats could easily have been compatible, the reversal of the order of the colors means that they are not – and worse still, a generic STL file reader cannot automatically distinguish between them. There is also no way to have facets be selectively transparent because there is no per-facet alpha value – although in the context of current rapid prototyping machinery, this is not important. </p>

      <h2><span class="mw-headline" id="The_facet_normal">The facet normal</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=5" title="Edit section: The facet normal">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>In both ASCII and binary versions of STL, the <a href="/wiki/Normal_(geometry)" title="Normal (geometry)">facet normal</a> should be a <a href="/wiki/Unit_vector" title="Unit vector">unit vector</a> pointing outwards from the solid object.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> In most software this may be set to (0,0,0), and the software will automatically calculate a normal based on the order of the triangle vertices using the "<a href="/wiki/Right-hand_rule" title="Right-hand rule">right-hand rule</a>", i.e. the vertices are listed in counter-clock-wise order from outside. Some STL loaders (e.g. the STL plugin for Art of Illusion) check that the normal in the file agrees with the normal they calculate using the right-hand rule and warn the user when it does not. Other software may ignore the facet normal entirely and use only the right-hand rule. Although it is rare to specify a normal that cannot be calculated using the right-hand rule, in order to be entirely portable, a file should both provide the facet normal and order the vertices appropriately. A notable exception is <a href="/wiki/SolidWorks" title="SolidWorks">SolidWorks</a>, which uses the normal for <a href="/wiki/Shading" title="Shading">shading effects</a>. </p>
      <h2><span class="mw-headline" id="Use_in_3D_printing">Use in 3D printing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=6" title="Edit section: Use in 3D printing">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <div class="thumb tright">
       <div class="thumbinner" style="width:222px;">
        <span class="mw-3d-wrapper" data-label="3D"><a href="/wiki/File:Wikipedia_puzzle_globe_3D_render.stl" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/b7/Wikipedia_puzzle_globe_3D_render.stl/220px-Wikipedia_puzzle_globe_3D_render.stl.png" decoding="async" width="220" height="165" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/b7/Wikipedia_puzzle_globe_3D_render.stl/330px-Wikipedia_puzzle_globe_3D_render.stl.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/b7/Wikipedia_puzzle_globe_3D_render.stl/440px-Wikipedia_puzzle_globe_3D_render.stl.png 2x" data-file-width="5120" data-file-height="2880"></a></span>
        <div class="thumbcaption">
         <div class="magnify">
          <a href="/wiki/File:Wikipedia_puzzle_globe_3D_render.stl" class="internal" title="Enlarge"></a>
         </div>Wikipedia logo
        </div>
       </div>
      </div>
      <p><a href="/wiki/3D_printing" title="3D printing">3D printers</a> build objects by solidifying (<a href="/wiki/Stereolithography" title="Stereolithography">SLA</a>, <a href="/wiki/Selective_laser_sintering" title="Selective laser sintering">SLS</a>, <a href="/wiki/Selective_heat_sintering" title="Selective heat sintering">SHS</a>, <a href="/wiki/Selective_laser_melting" title="Selective laser melting">DMLS</a>, <a href="/wiki/Electron-beam_additive_manufacturing" title="Electron-beam additive manufacturing">EBM</a>, <a href="/wiki/Digital_Light_Processing" title="Digital Light Processing">DLP</a>) or printing (3DP, MJM, <a href="/wiki/Fused_filament_fabrication" title="Fused filament fabrication">FDM</a>, <a href="/wiki/Fused_filament_fabrication" title="Fused filament fabrication">FFF</a>, PJP, MJS)<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup> one layer at a time. This requires a series of closed 2D contours (horizontal layers) that are filled in with solidified material as the layers are fused together. A natural file format for such a machine would be a series of closed polygons (layers or slices) corresponding to different Z-values. However, since it is possible to vary the layer thicknesses for a faster though less precise build, it was easier to define the model to be built as a closed <a href="/wiki/Polyhedron" title="Polyhedron">polyhedron</a> that can be sliced at the necessary horizontal levels. An incorrect facet normal can affect the way a file is sliced and filled. A slice at a different Z-value can be chosen to miss a bad facet or the file must be returned to CAD program to make corrections and then regenerate the STL file. </p>
      <p>The STL file format appears capable of defining a polyhedron with any polygonal facet, but in practice it is only ever used for triangles, which means that much of the syntax of the ASCII protocol is superfluous. </p>
      <p>To properly form a 3D volume, the surface represented by any STL files must be closed (no holes or reversed vector normal) and connected, where every edge is part of exactly two triangles, and not self-intersecting. Since the STL syntax does not enforce this property, it can be ignored for applications where the void does not matter. The missing surface only matters insofar as the software that slices the triangles requires it to ensure that the resulting 2D polygons are closed. Sometimes such software can be written to clean up small discrepancies by moving vertices that are close together so that they coincide. The results are not predictable, may be sufficient of the surface must be repaired using another program. Vector 3D printers require a clean .STL file and printing a bad data file will either fail to fill or may stop printing. </p>
      <h2><span class="mw-headline" id="Use_in_other_fields">Use in other fields</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=7" title="Edit section: Use in other fields">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <div class="thumb tright">
       <div class="thumbinner" style="width:252px;">
        <span class="mw-3d-wrapper" data-label="3D"><a href="/wiki/File:Utah_teapot_(solid).stl" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/Utah_teapot_%28solid%29.stl/250px-Utah_teapot_%28solid%29.stl.png" decoding="async" width="250" height="188" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/Utah_teapot_%28solid%29.stl/375px-Utah_teapot_%28solid%29.stl.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/93/Utah_teapot_%28solid%29.stl/500px-Utah_teapot_%28solid%29.stl.png 2x" data-file-width="5120" data-file-height="2880"></a></span>
        <div class="thumbcaption">
         <div class="magnify">
          <a href="/wiki/File:Utah_teapot_(solid).stl" class="internal" title="Enlarge"></a>
         </div><a class="external text" href="https://upload.wikimedia.org/wikipedia/commons/9/93/Utah_teapot_%28solid%29.stl">STL model</a> of the <a href="/wiki/Utah_teapot" title="Utah teapot">Utah teapot</a> viewed in the <a class="external text" href="https://mediawiki.org/wiki/Extension:3D">MediaWiki 3D extension</a>
        </div>

      <p>STL file format is simple and easy to output. Consequently, many <a href="/wiki/Computer-aided_design" title="Computer-aided design">computer-aided design</a> systems can output the STL file format. Although the output is simple to produce, mesh connectivity information is discarded because the identity of shared vertices is lost. </p>
      <p>Many <a href="/wiki/Computer-aided_manufacturing" title="Computer-aided manufacturing">computer-aided manufacturing</a> systems require triangulated models. STL format is not the most memory- and computationally efficient method for transferring this data, but STL is often used to import the triangulated geometry into the <a href="/wiki/Computer-aided_manufacturing" title="Computer-aided manufacturing">CAM</a> system. The format is commonly available, so the CAM system will use it. In order to use the data, the CAM system may have to reconstruct the connectivity. As STL files do not save the physical dimension of a unit, a CAM system will ask for it. Typical units are mm and inch. </p>
      <p>STL can also be used for interchanging data between CAD/CAM systems and computational environments such as <a href="/wiki/Mathematica" class="mw-redirect" title="Mathematica">Mathematica</a>. </p>
      <h2><span class="mw-headline" id="Representation_of_curved_surfaces">Representation of curved surfaces</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=8" title="Edit section: Representation of curved surfaces">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>It is not possible to use triangles to perfectly represent curved surfaces. To compensate, users often save enormous STL files to reduce the inaccuracy. However, native formats associated with many 3D design software use <a href="/wiki/Surface_(topology)" title="Surface (topology)">mathematical surfaces</a> to preserve detail losslessly in small files. For example, <a href="/wiki/Rhino_3d" class="mw-redirect" title="Rhino 3d">Rhino 3D</a><sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> and <a href="/wiki/Blender_(software)" title="Blender (software)">Blender</a><sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> implement <a href="/wiki/Nurbs" class="mw-redirect" title="Nurbs">NURBS</a> to create true curved surfaces and store them in their respective native file formats, but must generate a triangle mesh when exporting a model to the STL format. </p>
      <h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=STL_(file_format)&amp;action=edit&amp;section=9" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
      <p>STL was invented by the Albert Consulting Group for <a href="/wiki/3D_Systems" title="3D Systems">3D Systems</a> in 1987.<sup id="cite_ref-All3DP_13-0" class="reference"><a href="#cite_note-All3DP-13">[13]</a></sup> The format was developed for 3D Systems' first commercial 3D printers. Since its initial release, the format remained relatively unchanged for 22 years.<sup id="cite_ref-stl2_14-0" class="reference"><a href="#cite_note-stl2-14">[14]</a></sup> </p>
      <p>In 2009, an update to the format, dubbed STL 2.0, was proposed. It evolved into the <a href="/wiki/Additive_manufacturing_file_format" title="Additive manufacturing file format">Additive manufacturing file format</a>.<sup id="cite_ref-stl2_14-1" class="reference"><a href="#cite_note-stl2-14">[14]</a></sup><sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup> </p>
